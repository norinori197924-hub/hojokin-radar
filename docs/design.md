# データ取得頻度・取得件数の拡大 設計書

## 1. 現状確認結果

### 1.1 `.github/workflows/daily.yml`
- `cron: "0 21 * * *"`（JST 06:00）で1日1回実行。`workflow_dispatch`による手動実行も可能。
- 実行順: fetch_jgrants → fetch_jnet21(未実装ならスキップ) → enrich(未実装ならスキップ)
  → update_status → generate_site → commit&push。各fetchステップは
  `continue-on-error: true` で他ソースの失敗を波及させない設計。

### 1.2 `scripts/fetch_jgrants.py` の現状
- **取得件数の上限は実装上は存在しない**（`[:N]`等の打ち切りロジックは無い）。
  実質的な上限は「`SEARCH_KEYWORDS`（`補助`,`助成`,`支援`,`創業`,`事業`の5語）で
  ヒットする範囲」に限定されていること。
- `REQUEST_INTERVAL_SEC = 0.3`（秒）で全リクエストの間隔を空けている。

### 1.3 jGrants APIの仕様（公式ドキュメント・技術記事で確認）
- `GET /exp/v1/public/subsidies` は **page/limit/offsetなどのページング機構が存在しない**。
  `keyword`（2〜255文字、必須）で全文検索し、ヒットした全件を1レスポンスで返す方式。
  一覧を無条件に全件取得するAPIは無く、必ず何らかのkeywordを渡す必要がある。
- レート制限は**60リクエスト/分（推奨値）**、タイムアウトは30秒。
- → 現状の `REQUEST_INTERVAL_SEC = 0.3` は **約200リクエスト/分に相当し、
  公式推奨レートを超過している**（CAUTION、後述の修正で対応）。

### 1.4 実測によるキーワード網羅性の検証
実際にAPIを呼び出し、現行5キーワードと拡張候補15キーワードでの重複排除後の
ユニークid数を比較した（`acceptance=1`条件、2026-07-11時点）。

| キーワード数 | ユニークid数 |
|---|---|
| 現行5語（補助/助成/支援/創業/事業） | 275件（`data/subsidies.json`の現在件数と一致） |
| 上記+10語（中小企業/小規模/DX/IT/省エネ/人材/観光/ものづくり/女性/海外展開） | 299件（+24件, +8.7%） |

キーワードを増やすほど単調にヒット件数は増えるが、広い語（補助/助成/支援/事業）で
大部分をカバー済みのため、追加語による増分は逓減する。それでも取りこぼしている
個別分野（IT・観光・環境・人材等の専門語のみを含み「補助/助成/支援/事業」を
含まない案件）を拾うためにキーワードの拡充は有効。

## 2. 設計方針

### 2.1 「取得件数の上限撤廃」の解釈
APIにページング機構が無いため、"上限を撤廃してページネーションを実装する"という
対応は**技術的に不可能**（存在しないAPI機能は実装できない）。CLAUDE.mdの
「推測でデータを補完しない」原則に基づき、存在しないページングAPIを前提にした
実装は行わない。代わりに、キーワードカバレッジを広げることで実質的な取得件数を
拡大する。

- `SEARCH_KEYWORDS` を5語から、主要な支援分野を横断的にカバーする語彙に拡張する
  （IT・DX、ものづくり、観光、農業、環境・脱炭素、人材育成、事業承継、女性活躍、
  海外展開、賃上げ、スタートアップ等）。
- 拡張後もidベースで重複排除する既存ロジック（`ids[item["id"]] = item`）をそのまま
  踏襲する。

### 2.2 レート制限準拠
- `REQUEST_INTERVAL_SEC` を `0.3` → `1.1` に変更し、60リクエスト/分の推奨値を
  下回るようにする（安全マージンとして1.1秒間隔＝約54リクエスト/分）。
- `search_ids()` に個別キーワード単位のtry/exceptを追加する。現状は検索フェーズで
  1キーワードでも通信エラーになるとスクリプト全体が例外で停止する。キーワード数が
  増える分、単発の通信エラーで全体が失敗するリスクも増えるため、詳細取得フェーズ
  （`fetch_detail`呼び出し側）と同様にWARNログを出して次のキーワードへ継続する。

### 2.3 更新頻度についての判断
**結論: 日次のまま維持することを推奨する。**理由:

- jGrants側の公募情報は日単位で更新される性質のものであり、分・時間単位の鮮度は
  実用上不要（締切日という日付粒度のデータであり、`update_status.py`のstatus
  再計算も日付ベース）。
- 1日複数回実行しても「新着」検知が数時間早まる程度のメリットに対し、
  API呼び出し回数・GitHub Actions実行時間が実行回数分そのまま増える
  （後述2.4）。レート制限自体には抵触しないが、外部の公開APIへの負荷を
  不必要に増やすべきではない。
- 需要が生じた場合（例:「締切間近」の見逃しを減らしたい等）は、まず
  `daily.yml`に`workflow_dispatch`が既にあるため手動実行で対応し、恒常的な
  複数回実行は当面見送る。

### 2.4 ビルド時間・GitHub Actions実行時間の試算
- 現行: 5キーワード検索 + 275件詳細取得 ≒ 280リクエスト × 0.3秒 ≒ 84秒
  （+ ネットワーク往復時間）。
- 変更後: 約20キーワード検索 + 実測ベースで300〜350件程度の詳細取得を想定
  ≒ 320〜370リクエスト × 1.1秒 ≒ 6〜7分。
- GitHub Actionsのジョブタイムアウト（デフォルト最大6時間）に対して十分小さく、
  許容範囲。日次1回の実行であれば実行時間・Actions分数消費の増加は問題にならない
  と判断する。

### 2.5 重複データ・終了済み補助金の扱い（現状のまま維持）
現行の`fetch_jgrants.py`・`update_status.py`の設計は既にCLAUDE.mdの原則
（締切切れは削除せずstatus="closed"）に沿っており、変更不要と判断する。

- `fetch_jgrants.py`はid単位でupsertし、今回の検索でヒットしなかった既存項目
  （＝募集終了などで`acceptance=1`の対象から外れた項目）は`existing`辞書から
  削除されず、そのままdata/subsidies.jsonに残る。
- `status`は`update_status.py`が毎回、締切日と実行日を比較して
  `open`/`closing_soon`/`closed`に再計算するため、jGrants側の`acceptance`
  フラグの揺れ（CLAUDE.mdに記載の「締切超過でも1のままのことがある」）に
  依存しない。
- 重複は`id`（`jgrants-{id}`）をキーにしたdictで自動的に排除される。

## 3. 変更差分

- `scripts/fetch_jgrants.py`:
  - `SEARCH_KEYWORDS` を拡張（5語 → 約20語）
  - `REQUEST_INTERVAL_SEC` を `0.3` → `1.1` に変更
  - `search_ids()` に1キーワード単位のtry/exceptを追加（部分失敗時も継続）
- `.github/workflows/daily.yml`: 変更なし（日次のまま維持と判断）
- `data/subsidies.json` のスキーマ・命名規則: 変更なし
- `scripts/update_status.py` / `scripts/generate_site.py`: 変更なし
