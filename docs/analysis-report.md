# 分析レポート（データ取得頻度・取得件数の拡大）

担当: analyzer サブエージェント（GitHub Actions実行時間・API負荷観点。
今回はバックエンドのみの変更のためSEO/GA4は対象外）

## 1. GitHub Actions実行時間

- 通常想定: 検索21件+詳細300〜350件 × 1.1秒間隔 ≒ 約6〜7分（sleep時間ベースの
  理論下限）。実際のネットワーク応答時間を加えても数分〜十数分程度と推定され、
  許容範囲。
- ワーストケース（全リクエストが30秒タイムアウトまで待つ場合）: 理論上約3.2時間。
  GitHub Actions既定の6時間上限には収まるが、想定を大きく超える。

重大度: 低（通常運用では問題なし、ただし実測ログが無く要監視）

## 2. `timeout-minutes` 未設定

`daily.yml`の`update`ジョブに`timeout-minutes`が無く、既定6時間が上限。
`fetch_jgrants.py`ステップに`continue-on-error: true`が付いているため、
異常な長時間実行が起きてもワークフロー全体は失敗扱いにならず気づきにくい。

**推奨: `timeout-minutes: 30`を`update`ジョブに設定**（通常想定7〜15分に対し
約2倍の安全マージン、異常時は30分で早期検知）。

重大度: 中 → **今回修正**

## 3. jGrants公開APIへの負荷

- `REQUEST_INTERVAL_SEC=1.1`秒（約54.5リクエスト/分）は公式推奨レート制限
  60リクエスト/分に対し約10%のマージンを確保しており、日次1回のバッチ実行という
  頻度も踏まえ適正利用の範囲内。
- HTTP 429等のレート制限超過を専用にハンドリングするリトライ/バックオフが無く、
  抵触時はデータ欠落につながりうる（データを推測で埋めるわけではないため
  CLAUDE.mdの原則には反しないが、改善余地として記録）。

重大度: 低〜中 → 今回は見送り（将来的な改善候補として記録）
